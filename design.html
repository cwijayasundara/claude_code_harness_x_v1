Lean Expert-Generalist Harness

A deliberately small Claude Code plugin that turns a bounded request into a tested, reviewable draft PR. Humans approve meaning, tests, architecture, and material risk. Agents execute one story at a time—never marking their own homework.

1. Executive summary

Agent = model + harness. The model writes code; the harness is everything around it that makes output trustworthy—guides before action, sensors after, and a human steering loop that improves the system over time.

A senior engineer plans before coding, writes the test first, never marks their own homework, refuses silent quality slip, reviews before merge, and learns from every mistake. The harness encodes those habits as mechanisms—so behaviour is predictable, not aspirational.

What it is

Thin Claude Code plugin

Skills, agents, one safety hook, deterministic Node scripts and contracts. Target repos get a short CLAUDE.md; control plane stays under .claude/.

Core idea

Co-design then ratchet

Human-gated G0–G4 (and B0–B2 brownfield) freezes intent and shape. The story ratchet then enforces red → green → independent review → sensors → verified.

Guarantee

Generator ≠ evaluator

Sonnet implements; Opus evaluates read-only in fresh context. Deterministic sensors never trust agent narrative. Merge and deploy stay human.

Outcome

Draft PR readiness

Auditable specs, evidence hashes, branch review, and fail-closed pre-PR verification—not automatic merge, not a second Devin.

2. Problem & motivation

Left unharnessed—or harnessed with unbounded ceremony—agentic coding hits failure modes that compound across sessions and report types.

2.1 Three classic agent failures

Self-evaluation bias

Author grades own work

Confirmation bias, rationalizing failing tests, scope creep. Defeated by: independent evaluator + real command evidence.

Quality regression

Codebase decays per story

God files, layer violations, copy-paste pipelines, silent gate bugs. Defeated by: boundaries, file-size, duplication sensors, evolutionary G3.

Context exhaustion

Sessions lose the plot

Work restarts; requirements compress; mistakes repeat. Defeated by: immutable specs, story contracts, protected context packs, feature-branch checkpoints.

2.2 The v5 trap (why this product is lean)

Engine v5 growth mode

  • ~49 skills, many agents, 12+ hook surfaces
  • ~138 scripts / ~19.5k LOC in scripts alone
  • Gap ledger G1–G36; every article → new control
  • Simplification proposals that still added surface
  • Public command explosion

Lean harness reaction

  • Three public delivery skills + operations
  • Three agents; one destructive-Git hook
  • Deterministic contract spine in Node
  • Control budget + net-add justification
  • Retro may remove zero-fire sensors

3. Doctrine

Non-negotiables that every design choice must pass.

  1. Humans steer product meaning — approvals recorded with name and time in index.json.
  2. Repository is system of record — if the agent cannot open it, it does not exist.
  3. Specs before code — no product implementation before G4 (and B2 when brownfield).
  4. Tests before green — red evidence is a real command with non-zero exit, not invented.
  5. Generator never self-advances the ratchet — main workflow owns state transitions.
  6. Evaluator is fresh-context and read-only — never trusts generator narrative.
  7. One automated repair — then human-decision-required.
  8. Draft PR only — no merge, deploy, or fleet automation.
  9. Protected branches are write-forbidden for specs and product code (main/master/develop).
  10. Subtractive ratchet on the harness — every new control pays rent or replaces another.

4. Architecture: two loops only

Complexity is capped by refusing a third loop. Everything is either co-design (no product code) or story execution (bounded, evidenced).

Loop 1 · Co-design

Human-gated · no product code

G0→G4 greenfield; B0→B2 then G1–G4 brownfield. Agent drafts; human revises; proposal packs are reviewable markdown sessions; approve freezes artifacts.

Loop 2 · Story ratchet

Agent executes · sensors + evaluator

One dependency-ready story at a time. RED_TEST → IMPLEMENT → STORY_REVIEW → FAST_SENSORS → STORY_VERIFIED. Then pre-PR + branch review → draft PR readiness.

End-to-end greenfield

G0→ G1→ G2→ G3→ G4→ Ratchet × N→ Pre-PR→ Branch review→ Draft PR

5. System layers

Progressive disclosure: load only what the task needs.

Base craft

TDD, debugging, cohesion, dependency direction, pragmatic SOLID/OO, DDD where approved, refactoring, measured performance — skill harness-engineering-core.

Language packs

Path-scoped Python / TypeScript guides + sensors. Loaded by signal, not every session.

Framework packs

Optional FastAPI, LangChain, LangGraph, Deep Agents, Google ADK — only when profiles match. Never a framework encyclopedia in always-on context.

Outer control

Spec packages, human gates, story ratchet, sensors, cost routing, verification plan, draft-PR readiness, control budget.

6. Co-design gates (G0–G4)

Superpowers-style design sessions: agent drafts, human decides. Each gate produces a narrative proposal pack (not only raw JSON) before approve.

GatePackagesHuman decides
G0 source + brd/prd Grounded interpretation: outcomes, in/out scope, assumptions, open questions
G1 epics, stories, dependencies Vertical stories with acceptance; dependency DAG / sequence
G2 test-data, test-cases, test-plans Cases, fixtures, hermetic strategy, doubles, expected results
G3 design, architecture Seams, folder layout, structural shape, budgets, second-slice policy
G4 plans (story contracts) Scope, tests, sensors, implementation_posture, reuse targets
Proposal UX. harness-specs.js proposal --change … --gate Gx --write writes .claude/specs/evidence/*-gate-Gx-proposal.md with a human decision session (checklist + narrative) and a JSON appendix. Do not ask humans to approve raw JSON alone.

Gate sequence rules

  • Gates are sequential; predecessor approval is required.
  • Approved artifacts are immutable; change requires amendment + re-approval.
  • Ambiguity that affects behaviour, security, architecture, or tests → stop and ask.

7. Evolutionary architecture (G3 + G4)

Prevents the “six cloned pipelines” failure mode without baking in any framework spine (LangGraph, workers, plain modules — project choice).

7.1 Three structural alternatives (required at G3)

clone-vertical

Independent stacks

Full vertical slice per feature/entity. Fast first delivery; highest duplication risk.

shared-modules

Thin entrypoints + helpers

Shared pure helpers without a single orchestrating spine. Good when flows diverge but helpers do not.

parameterized-spine

One flow + strategies

Stages parameterized by config/strategy. Prefer when skeletons match. Technology is chosen after shape.

Architecture must also set selected_alternative_id, selection_rationale, second_slice_reuse_policy, and evolutionary_rules. Enforced by lib/design-evolution.js on proposal and approve.

7.2 Implementation posture (required at G4)

PostureWhenRequires
first-sliceFirst capability; empty depsMulti-story independent slices need explicit human decision text
reuse-existingCall proven helpers/seamsNon-empty reuse_targets
extract-sharedPay for a shared abstractionReuse targets + design refs
justified-divergenceTruly different algorithmsdivergence_justification

Dependent stories cannot claim first-slice. Multi-story canary proves: story-1 first-slice → story-2 reuse-existing; bad first-slice on story-2 is blocked.

8. Story ratchet

Monotonic story progress with hard evidence. Quality cannot silently slip past a state gate.

READY→ RED_TEST→ IMPLEMENT→ STORY_REVIEW→ FAST_SENSORS→ STORY_VERIFIED
CheckEnforcement
Feature branch + G4 approvedStart refused otherwise
Dependency stories verifiedRatchet dependency check
Red: real failing testCommand, exit ≠ 0, expected vs observed
Green: real passing testExit 0; paths within allowed_change_scope
Independent reviewpass | revise | human-decision-required
Repair budgetOne automated repair; then human
SensorsNormalized report recorded before verify
Evidence integrityHash / freshness; tampered evidence fails verify
State is not proof. .claude/state/stories/ is resumable progress only. Correctness lives in evidence + approved specs under .claude/specs/.

9. Brownfield path (B0–B2)

Map before change. Reuse first. Generalize only with ≥2 real uses.

B0 baseline→ B1 code map→ B2 strategy→ G1–G4→ Ratchet
  • B0 — Record exact baseline build/test/lint/type/security; do not blame the change for pre-existing red.
  • B1 — Bounded map with provenance (AST/LSP and optional Graphify/CCE adapter). Canonical reuse + duplicate candidates. Open cited source before claiming.
  • B2 — Smallest seam, second_slice_decision, reuse targets or divergence justification, characterization tests, regression scope, design amendment.

Lived brownfield canary fails if required reuse is omitted and requires calling the existing helper rather than re-copying logic.

10. Generator ≠ evaluator (GAN separation)

Inspiration from Engine v5 white paper: separation of writer and grader is structural, not a prompt suggestion.

harness-generator

Sonnet · implement

Receives contract + context pack only. Finds canonical local pattern. Writes tests and code. Cannot advance ratchet state.

harness-evaluator

Opus · read-only

Fresh context. Sources, design, story, tests, diff, evidence. Strict JSON verdict. Never sees generator’s suggested pass.

harness-evaluator-fast

Haiku · off by default

Economical path only after matched pilots preserve quality and a human enables promotion. Strong risks never use it.

11. Sensors & local verification

Fowler-style guides (feedforward) + sensors (feedback). Prefer computational sensors every story; inferential review where semantics matter. Agent-consumable: status, paths, reason, correction.

11.1 Built-in computational sensors

SensorDefaultPurpose
secret-scanfail on hitCredential patterns in scope
architecture-boundariesfail on violationExecutable layer rules in boundaries.json
file-sizemax 300 lines · warn 250God files / unbounded routers
near-duplicationwarn · ≥8 significant linesCopy-paste-and-adapt clusters

Thresholds: .claude/project/maintainability.json. Profile sensors add lint/type/test/SAST by stack.

11.2 Pre-PR local engineer simulation

Target-owned verification.json must configure real commands for:

  • install/build, unit, hermetic integration, hermetic system regression
  • public-seam smoke success and failure journeys
  • lint, type, security, contract/boundary doubles, performance budgets
Fail-closed. Unconfigured required kinds throw before execution. No “tests green because we forgot to configure them.”

11.3 Control budget (anti-v5)

Every active control in harness-manifest.json declares direction, execution type, regulates axes, owner, recurring failure, cost, review date, removal path, and (for sensors) self-correction. Budget: max_active + baseline_ids; net-adds need justification or replaces.

12. Routing, context & cost

  • Deterministic first — scripts need no model.
  • Sonnet — bounded implementation sidekick.
  • Opus — architecture, domain, security, privacy, public-contract, migration, performance, branch review.
  • Context packs — role budgets; requirements, approvals, invariants, expected results never compressed; only verbose tool output may be excerpted with hash/line provenance.
  • Cost receipts — provider/session evidence only; never invent tokens or dollars.

15. Init & upgrade

  • harness-init — create missing scaffold files only; never overwrite target CLAUDE.md or policy.
  • harness-upgrade --apply — missing files + merge new baseline controls (file-size, near-duplication) into existing manifests; create maintainability.json if absent; never clobber custom control fields.
node "$CLAUDE_PLUGIN_ROOT/scripts/harness-upgrade.js" --target /path/to/app --apply
node "$CLAUDE_PLUGIN_ROOT/scripts/harness-validate.js" /path/to/app

16. Relationship to Claude Harness Engine v5

This design keeps v5’s ideas (GAN, ratchet, humans on the loop, sensors) and rejects v5’s platform bloat. The white paper’s thesis remains; the product surface is an order of magnitude smaller.

17. Out of scope forever (this product)

  • Dashboard, scheduler, fleet management
  • Automatic merge or deploy orchestration
  • General workflow engine / multi-SKU packaging
  • Linear/Jira platform as core (optional project MCP only)
  • Gap ledgers as product (G1–G36 style)
  • Rebuilding Claude Code’s agent runtime or permission system

18. Success metrics

Measure accepted story quality and review toil—not control count or token vanity.

  • First-pass story acceptance rate
  • Human review minutes per accepted story
  • Escaped defects after draft PR
  • Sensor precision and correction rate
  • Provider cost per accepted story (from real receipts)
  • Brownfield map usefulness (reuse hit rate)

Synthetic canaries prove deterministic integration. Model quality and human time remain pilot/human-gated (M7 scorecard never auto-rollouts).

Appendix A · Industry synthesis (kept)

  • Fowler — guides + sensors; computational first; harness improvement over line-by-line review
  • SPDD / artifact DSL — structured, linkable specs; control grounding not a new language
  • OpenAI Codex harness — progressive disclosure; enforce architecture mechanically
  • Anthropic long-running agents — initializer then incremental worker; feature list with pass flags
  • Stripe Minions — path-scoped checks; cap repair rounds
  • Devin/Cognition — plan before autonomy; pre-agreed tests; status only (no full rebuild)
